Some Background

What exactly is Continuous Delivery?

  1. CDel=full UT and FT on every checkin + metrics that you trust.
  2. Automated FTs, no manual tests
  3. Build Pipeline; code passes & Prod-ready or kicked out of the the pipeline

Some Background

What exactly is Continuous Delivery?

... so then Continuous Integration??

  1. CInt=compile+UT+metrics on every checkin
  2. So CDel = CInt+funtional tests; Eligible for Prod, but not mandatory to go

Some Background

What exactly is Continuous Delivery?

... so then Continuous Integration??

... Fine, but WTF is Continuous Deployment?!?

  1. CDep=CDel+actually deploying to prod on *every* passing automated CDel build

Why Should You Care?

  1. When PO goes "hey that feature we finished coding today". No all-nighter.
  2. And trust me, they will do things like that to you. And being ready for it feels awesome.
  3. Waiting in test regions for people to review and signoff on code sucks.
  4. Allows you to fail fast. If an idea you have doesn't work well, you know it sooner. That is good.

How is PayPal Credit doing?

  1. We were always a CI shop, but FT was manual.
  2. Starting with new services is much easier than backporting old services.
  3. At this point all active development services are CDel.
  4. Many teams deploy after each sprint; trying to go faster without getting sloppy
  5. We do have teams within PayPal deploying 2-3 times per day

Wrapping up

  1. Read the book
  2. When you go start your own company, build CDel or CDep in from the start. Make it part of the base you build on.
  3. Take your tests seriously, fix them immediately when failed. Never allow a release from a build with failed tests.
  4. Monitor your test coverage, high coverage gives warm and fuzzy about the safety of releasing to Prod.
